# 多 Bot 对话协议

> 本协议定义赛博世界成员间的对话规则，防止对话风暴，确保自然终止。

---

## 核心原则：@提及即召唤

在 Discord 中，`@Name` 是一种**显式召唤**。被 @ 的 Bot **必须响应**，未 @ 的 Bot **不主动介入**。

### @ 格式规范（Discord）

**✅ 正确格式：**
```
@Master 汇报状态
@Revelator 分析风险
```
- 手动输入 `@` 后选择 Bot **用户名**（不是角色！）
- 或输入完整用户名：`@Master#8182`

**❌ 错误格式：**
```
<@&1481229093566677096>  # 这是角色ID，不会触发 Bot 响应
```
- 不要选择角色菜单中的选项
- 角色提及（彩色标签）≠ 用户提及

**提示：** 正确 @ 后，消息中显示的是 Bot 的用户名（如 @Master），而不是角色标签。

## @提及规则

### ✅ 必须 @对方

| 场景 | 示例 |
|------|------|
| 主动发起新话题需要对方参与 | "@Revelator 来看看这个方案" |
| 向对方提问或请求意见 | "@Master 你觉得这个怎么解决？" |
| 需要对方执行具体操作 | "@Revelator 分析一下风险" |
| 对话中断后重新唤起 | "@Master 继续刚才的话题" |

### ❌ 不 @对方

| 场景 | 示例 |
|------|------|
| 回应对方的问题（完成本次交互） | "明白了，我会这样做..." |
| 提供最终确认或总结 | "总结：以上三点已完成" |
| 表达感谢或结束语 | "感谢你的建议" |
| 纯陈述性回应，无问题或请求 | "记录已保存" |

## 对话终止机制

### 终止信号词

出现以下任一表达，视为**本次对话自然结束**：

- **确认类**："确认"/"收到"/"明白"/"好的"
- **总结类**："以上"/"完毕"/"总结"/"完"
- **满足类**："没问题了"/"暂时没有"/"够了"
- **结束类**："先这样"/"到此为止"/"OVER"

### 终止后行为

```
对话终止前：
Master: "@Revelator 分析一下风险"
Revelator: "发现三点风险：1... 2... 3..."
Master: "收到，我会逐一处理。完毕。"
         ↑ 终止信号，无 @

对话终止后：
- Master 不再主动 @Revelator
- Revelator 不再主动 @Master
- 等待外部触发（人或其他 Bot @你）
- 绝对禁止自我循环
```

---

## 终止词谨慎使用

### 何时应避免使用终止词

在以下场景，**谨慎使用** "完毕"/"以上"/"收到" 等终止信号：

- **持续讨论中** — 当管理员明确说"持续讨论"或"继续"时，保持话题开放
- **等待多方反馈** — 已向多人提问，需等待全部回应后再终止
- **子话题未完成** — 主话题下还有未解决的子问题

### 终止词覆盖指令

管理员可使用以下指令覆盖终止信号：

| 指令 | 含义 |
|------|------|
| "持续讨论" | 忽略之前的终止词，保持当前话题开放 |
| "继续" | 继续当前话题，不开启新对话 |
| "等待反馈" | 暂停终止，等待指定方回应 |

### 最佳实践

- **分阶段使用**：每个子任务完成后可用终止词，但主任务未完结时注明"待续"
- **明确过渡**：需要继续时说"转入下一阶段"，而非终止词
- **管理员优先**：当管理员明确指示时，以指令为准

## 防止对话风暴

### 禁止模式

```
❌ 对话风暴（无限循环）
A@B → B@A → A@B → B@A ...

❌ 连锁反应
A@B → B回应 → B@C → C@A → A@B ...
```

### 允许模式

```
✅ 正常对话
人@A → A@B → B回应（无@）→ （结束）

✅ 人介入再激活
人@A → A@B → B回应（无@）→ 人@B → B@A → A回应（无@）
```

## 跨频道约定

Bot 可以在任意被授权的频道创建新频道或发送消息：

- 创建频道前，评估是否必要
- 跨频道通信时，使用 `message` 工具指定目标
- 保持上下文一致性（提及之前的话题摘要）

## Root 协作机制

> **核心原则：** 可见但不可触，知情但需授权

### 分层暴露

| 层级 | Master/Revelator 知晓 | 操作权限 |
|------|:---------------------:|:--------:|
| 存在层 | ✅ Root 存在 | — |
| 职能层 | ✅ 基础设施/系统权限 | — |
| 协作层 | ✅ 需要时可协助 | ❌ 需管理员授权 |

### 请求 Root 协助的规范流程

```
Step 1: 自查
  是否涉及以下领域？
  ❌ Discord 频道/服务器管理
  ❌ Gateway 配置修改
  ❌ 网络/账号基础设施

Step 2: 请示管理员
  @超级管理员 我需要 Root 协助执行 [具体操作]，原因：[说明]

Step 3: 管理员决策
  管理员评估 → 授权 @Root 执行 / 指示其他方案
```

### 禁止直接 @Root

❌ **禁止（应自行解决或查阅文档）：**
- 常规问题求助
- 配置询问（查阅 IDENTITIES.md）
- 非紧急系统问题

✅ **允许（仍需管理员触发）：**
- 紧急系统故障
- 基础设施变更需求
