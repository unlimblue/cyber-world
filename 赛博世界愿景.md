# 赛博世界愿景

## 愿景

我希望做一个**通用的**赛博世界框架，它可以被实例化为任意的组织形态，如：

- 🏛️ **封建王朝** —— 皇帝、丞相、太尉，等级森严的治理体系
- 🏢 **科技公司** —— CEO、CTO、产品经理，现代企业管理结构  
- 🎰 **地下赌场** —— 庄家、荷官、保安，灰色产业的运作机制
- 🏫 **魔法学院** —— 院长、教授、学生，奇幻世界的师徒传承
- 🏴‍☠️ **海盗船队** —— 船长、大副、水手，冒险者的松散联盟
- ...以及任何你能想象的组织形态

**核心思想**：框架提供通用的 Multi-Agent 协作基础设施，具体的角色、规则、流程由配置文件和 Prompt 定义，无需改动代码即可创造全新的赛博世界。

## 原则与操作规范

### 三阶段工作法

```
🔍 调研 → 📐 设计 → 🔨 实现
```

**每次只聚焦一个阶段！** 禁止跨阶段提前行动。

| 阶段 | 输出物 | 禁止事项 |
|------|--------|---------|
| **🔍 调研** | 调研报告、问题分析、需求收集 | 写设计文档、碰代码 |
| **📐 设计** | 架构文档、接口定义、流程图 | 写实现代码、配环境 |
| **🔨 实现** | 代码、测试、部署脚本 | 改设计文档（除非发现设计缺陷） |

### 目录结构规范

所有任务必须在 `archive/` 目录下按规范存储：

```
archive/
├── 2026-03-06-任务名称/          # 任务目录命名格式
│   ├── research/                 # 🔍 调研阶段文档
│   │   └── README.md            # 调研报告
│   ├── design/                   # 📐 设计阶段文档
│   │   └── README.md            # 设计文档
│   └── implementation/           # 🔨 实现阶段文档
│       └── README.md            # 实现记录
```

**命名规范**：`YYYY-MM-DD-任务名称`

### 阶段转换检查清单

#### 🔍 → 📐 调研转设计
- [ ] 调研报告已完成并归档
- [ ] 问题分析清晰明确
- [ ] 需求已收集整理
- [ ] 得到进入设计阶段的明确指令

#### 📐 → 🔨 设计转实现
- [ ] 架构设计文档已完成
- [ ] 接口定义已明确
- [ ] 数据结构设计完成
- [ ] 流程图已绘制
- [ ] 得到进入实现阶段的明确指令

### 当前任务记录规范

**当前任务** 必须在愿景文件中明确记录：

```markdown

**任务名称**: [具体名称]
**阶段**: [🔍调研 / 📐设计 / 🔨实现]
**开始日期**: [YYYY-MM-DD]
**目标**: [一句话描述目标]

### 交付物
- [ ] 交付物1
- [ ] 交付物2

### 相关归档
- 调研: `archive/YYYY-MM-DD-任务名称/research/`
- 设计: `archive/YYYY-MM-DD-任务名称/design/`
- 实现: `archive/YYYY-MM-DD-任务名称/implementation/`
```

### 任务完成归档流程

当任务完成时：

1. 将当前任务内容移动到 `archive/YYYY-MM-DD-任务名称/` 对应阶段目录
2. 更新愿景文件的"当前任务"部分为新任务
3. 提交 Git 并推送
